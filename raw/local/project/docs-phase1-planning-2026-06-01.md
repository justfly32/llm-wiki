# Phase 1 기획 문서: DB 온톨로지 분석 프로젝트

## 1. 프로젝트 개요

### 목적
내부에 산재한 수많은 DB의 테이블/필드 간 연관관계를 자동으로 탐지하고,
온톨로지(지식 그래프)로 구조화하여 데이터 자산의 가시성을 높인다.

### 참고 모델: 팔란티어 온톨로지
- **Ontology**: 조직의 운영 레이어이자 디지털 트윈
- **Semantic Layer**: Object Type(엔티티), Property(속성), Link Type(관계)
- **Kinetic Layer**: Action Type(업무 로직), Function(비즈니스 로직)
- **특징**: 데이터 + 로직 + 액션 + 보안을 하나의 레이어에서 통합 관리
- **핵심**: 원본 DB 스키마를 온톨로지 오브젝트로 매핑(Hydration)하여
         비즈니스 사용자가 데이터 엔지니어 없이도 데이터를 탐색·활용

## 2. 연관관계 유형 정의

### 유형 1: 구조적 관계 (Structural)
| 탐지 방법 | 설명 | 신뢰도 |
|-----------|------|--------|
| 외래키 (FK) | INFORMATION_SCHEMA에서 FK 제약조건 추출 | 0.95 |
| 인덱스 기반 | 동일 컬럼명 + 인덱스 패턴 매칭 | 0.70 |
| JOIN 패턴 | 쿼리 로그 분석으로 실제 JOIN 관계 탐지 | 0.85 |

### 유형 2: 명명 패턴 관계 (Naming)
| 탐지 방법 | 설명 | 신뢰도 |
|-----------|------|--------|
| 정확 매칭 | `user_id` == `user_id` | 0.90 |
| 정규화 매칭 | `userId` ≈ `user_id` ≈ `usr_id` | 0.75 |
| 접두사/접미사 | `order_id`, `order_date`, `order_status` → order 도메인 | 0.60 |
| 동의어 | `customer` ≈ `client` ≈ `buyer` | 0.50 |

### 유형 3: 데이터 유사도 관계 (Data Similarity)
| 탐지 방법 | 설명 | 신뢰도 |
|-----------|------|--------|
| 값 중복도 | 샘플 데이터의 실제 값 집합 비교 (Jaccard) | 0.80 |
| 패턴 매칭 | 이메일, 전화번호, 주민번호 등 형식 패턴 | 0.85 |
| 분포 유사도 | 데이터 타입, 길이, NULL 비율 등 메타 비교 | 0.65 |

### 유형 4: 의미적 관계 (Semantic / LLM)
| 탐지 방법 | 설명 | 신뢰도 |
|-----------|------|--------|
| LLM 추론 | 필드명 + 설명 + 샘플 → 의미적 관계 판단 | 0.70 |
| 도메인 분류 | 필드 패턴으로 도메인 자동 분류 (인물/주문/상품) | 0.60 |
| 업무 로직 추론 | 테이블명 + 컬럼명 → 업무 엔티티 추론 | 0.55 |

### 통합 신뢰도 점수
```
final_score = Σ(weight_i × score_i) / Σ(weight_i)
- FK 발견 시: 자동 채택 (score ≥ 0.95)
- 0.70 이상: 자동 제안 (사용자 확인)
- 0.50~0.70: 수동 검토 대상
- 0.50 미만: 무시
```

## 3. 기술 스택 선정

### 백엔드
| 영역 | 기술 | 선정 이유 |
|------|------|----------|
| 언어 | Python 3.11+ | 데이터 분석 생태계 풍부 |
| 그래프 DB | Neo4j (또는 NetworkX) | 온톨로지 그래프 저장/탐색 |
| API 서버 | FastAPI | 고성능 비동기, 자동 문서화 |
| 작업 관리 | Celery + Redis | 비동기 스키마 수집/분석 |
| LLM | OpenRouter API | 다양한 모델 활용 |

### 프론트엔드
| 영역 | 기술 | 선정 이유 |
|------|------|----------|
| 그래프 시각화 | Cytoscape.js | 대규모 그래프 성능 우수 |
| UI 프레임워크 | React + Tailwind | 컴포넌트 기반 개발 |
| 테이블/검색 | AG Grid | 대규모 데이터 테이블 |

### 인프라
| 영역 | 기술 | 선정 이유 |
|------|------|----------|
| 컨테이너 | Docker + Docker Compose | 로컬 개발 환경 |
| 오케스트레이션 | Kubernetes | 프로덕션 배포 |
| 모니터링 | Prometheus + Grafana | 메트릭 수집/시각화 |

## 4. 유사 오픈소스 프로젝트 조사

### Apache Atlas
- **특징**: Hadoop 생태계 중심, 메타데이터 관리 + 거버넌스
- **강점**: 데이터 계보(lineage) 추적, 분류 체계
- **약점**: 복잡한 설정, Hadoop 종속성

### DataHub (LinkedIn)
- **특징**: 현대적 데이터 스택 중심, 메타데이터 플랫폼
- **강점**: 다양한 커넥터, 스키마 변경 감지, 데이터 계보
- **약점**: 대규모 인프라 필요

### Amundsen (Lyft)
- **특징**: 데이터 검색/발견 중심, 경량
- **강점**: 간단한 설치, 직관적 UI
- **약점**: 온톨로지/관계 분석 기능 부족

### OpenMetadata
- **특징**: 통합 메타데이터 플랫폼
- **강점**: 데이터 품질, 협업 기능
- **약점**: 아직 성숙도 부족

### Neo4j
- **특징**: 네이티브 그래프 DB
- **강점**: Cypher 쿼리, 대규모 그래프 성능
- **약점**: 엔터프라이즈 라이선스 비용
